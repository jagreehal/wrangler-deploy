Both `wrangler-deploy` and the short alias `wd` are available after install.

## `wd init`

Scan local wrangler configs and generate `wrangler-deploy.config.ts` from what it finds in your filesystem.

## `wd introspect [--filter <prefix>] [--dry-run]`

Scan your live Cloudflare account and generate `wrangler-deploy.config.ts` from existing resources. Lists all D1 databases, KV namespaces, Queues, R2 buckets, Hyperdrive configs, and Vectorize indexes in your account.

If `CLOUDFLARE_API_TOKEN` is set, it also discovers Workers and their bindings, wiring up which worker uses which resource automatically. Without the token, it uses `wrangler login` credentials to list resources (but cannot fetch worker bindings).

`--filter <prefix>` limits results to resources and workers whose names start with the given prefix. Useful for large accounts with many unrelated resources.

`--dry-run` prints the generated config to stdout without writing the file.

```bash
# Using wrangler login (no token needed)
wd introspect

# With full worker binding discovery
CLOUDFLARE_API_TOKEN=your-token wd introspect

# Preview without writing
wd introspect --dry-run

# Only resources starting with "payments-"
wd introspect --filter payments-
```

## `wd plan --stage <name>`

Dry-run. Shows what resources would be created, are in sync, drifted, or orphaned.

## `wd apply --stage <name> [--database-url <url>]`

Provision resources. `--database-url` required for Hyperdrive on first apply. Idempotent and resumable.

## `wd deploy --stage <name> [--verify]`

Deploy workers using rendered configs. `--verify` runs post-deploy coherence checks. Blocks if secrets are missing.

## `wd destroy --stage <name> [--force]`

Tear down all resources. Requires `--force` for protected stages. Removes queue consumers before workers.

## `wd status [--stage <name>]`

Without stage: list all known stages. With stage: show resources, workers, drift.

## `wd verify --stage <name>`

Read-only coherence check. Validates state, rendered configs, resources, workers, secrets, service bindings.

## `wd secrets --stage <name>`

Check which declared secrets are set or missing.

## `wd secrets set --stage <name>`

Set missing secrets interactively.

## `wd secrets sync --to <name> --from-env-file <path>`

Bulk set secrets from a `.dev.vars`-style file.

## `wd gc`

Destroy stages past their TTL. Only affects unprotected stages.

## `wd graph [--stage <name>] [--format ascii|mermaid|dot|json]`

Show the topology graph of workers, resources, and bindings. Default format is `ascii`.

When `--stage` is provided, overlays live state (resource IDs, worker URLs, sync status).

```bash
wd graph                          # ASCII to terminal
wd graph --format mermaid         # Mermaid for Markdown/PRs
wd graph --stage staging          # Include live state
wd graph --format json            # Machine-readable
```

## `wd impact <worker-path>`

Show upstream dependencies, downstream dependents, and consequences if the worker is unavailable.

```bash
wd impact workers/api
```

## `wd diff <stage-a> <stage-b> [--format json]`

Compare two stages. Shows resource, worker, and secret differences.

```bash
wd diff staging production
wd diff staging production --format json
```

## `wd dev [--filter <worker>] [--port <base>]`

Start all workers in local dev mode. Automatically resolves available dev and inspector ports so multi-worker setups work without conflicts.

`--filter` starts only the target worker and its transitive service-binding dependencies.

```bash
wd dev                            # Start all workers
wd dev --filter workers/api       # Start api + its deps
wd dev --port 9000                # Override base port
```

## `wd ci init [--provider github] [--branch main]`

Generate a GitHub Actions workflow file at `.github/workflows/wrangler-deploy.yml`. The generated workflow includes apply, deploy, PR comments, check runs, cleanup on PR close, and production deploy on push to main.

```bash
wd ci init
wd ci init --branch main
```

## `wd ci comment --stage <name>`

Post or update a PR comment with deployment status, topology diagram, resource table, and worker URLs. Uses the `<!-- wrangler-deploy -->` marker to update in place on subsequent runs.

Requires `GITHUB_TOKEN` environment variable.

## `wd ci check --stage <name>`

Post a GitHub check run with success/failure status based on whether the stage has been deployed. Exits with code 1 if no state is found.

Requires `GITHUB_TOKEN` environment variable.

## `wd doctor`

Run diagnostic checks: wrangler version, authentication, config validation, and worker config existence.

```bash
wd doctor
```

## `wd completions --shell zsh|bash|fish`

Generate shell completion scripts. Pipe into a file or eval directly.

```bash
# Zsh
wd completions --shell zsh > ~/.zfunc/_wd

# Bash
wd completions --shell bash > /etc/bash_completion.d/wd

# Fish
wd completions --shell fish > ~/.config/fish/completions/wd.fish
```
