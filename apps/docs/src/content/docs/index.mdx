<span class="section-label">Local control plane</span>
<h2 class="section-title">Not just deploys. A live dashboard for the whole runtime.</h2>
<p class="section-desc">
  `wd dev ui` puts project defaults, agent metadata, workers, queues, D1, verify packs, snapshots, and logs into one repo-aware control plane. It is the fastest way to see what the CLI knows before you run the next command.
</p>

<div class="dashboard-showcase">
  <div class="dashboard-copy">
    <div class="feature-rows">
      <div class="feature-cell">
        <h3>Project defaults at a glance</h3>
        <p>The dashboard surfaces the resolved `.wdrc` / `.wdrc.json` values so agents and humans can see the active stage, ports, and auth-related defaults before they change anything.</p>
      </div>
      <div class="feature-cell">
        <h3>Command metadata built in</h3>
        <p>It reuses the same CLI manifest that powers `wd schema` and `wd tools`, so the UI can reflect the real command surface instead of a hand-written summary.</p>
      </div>
      <div class="feature-cell">
        <h3>Dev workflows, not just deploys</h3>
        <p>From endpoint calls and queue sends to D1 seed/reset flows and verify packs, `wd dev ui` keeps the runtime loop visible in one place.</p>
      </div>

<span class="section-label">What you get</span>
<h2 class="section-title">A better deploy workflow without throwing away your Wrangler workflow.</h2>

<div class="feature-rows">
  <div class="feature-cell">
    <h3>JSONC stays first-class</h3>
    <p>Your existing <code>wrangler.jsonc</code> files stay in the repo and stay readable. No rewrite step, no migration away from Wrangler. <code>wd init</code> reads them, <code>wrangler dev</code> still uses them, and <code>wd deploy</code> uses rendered JSONC built for the target stage. The checked-in JSONC remains the thing you author; wrangler-deploy adds orchestration around it.</p>
  </div>
  <div class="feature-cell">
    <h3>One tool for dev and deploy</h3>
    <p>Keep using Wrangler locally, or run <code>wd dev</code> to start every worker together with sane ports and dependency order. When you already applied a stage, <code>wd dev --stage staging</code> can reuse rendered bindings so local runtime matches deploy-time state. When you need Queue local-dev parity, switch to <code>wd dev --session --persist-to .wrangler/state</code>. Add <code>wd dev doctor</code>, <code>wd cron trigger</code>, and <code>wd queue list</code> when you need runtime visibility on top. The benefit is not replacing Wrangler's primitives. The benefit is turning them into repo-aware workflows so developers stop guessing which worker, which port, and which local route they need.</p>
  </div>
  <div class="feature-cell">
    <h3>Typed <code>Env</code> from config</h3>
    <p>One line — <code>type Env = typeof api.Env</code> — gives you <code>D1Database</code>, <code>KVNamespace</code>, <code>Queue&lt;T&gt;</code>, and <code>Fetcher</code> at compile time. No generated files, no sync step. Rename a binding in config and TypeScript finds every mismatch across every worker.</p>
  </div>
  <div class="feature-cell">
    <h3>Shared state across the team</h3>
    <p>Store state in a Cloudflare KV namespace and stop relying on one laptop having the latest stage metadata. One person can apply, another can deploy, CI can verify or destroy, and everyone sees the same state.</p>
  </div>
  <div class="feature-cell">
    <h3>Safer staged deploys</h3>
    <p>Protected stages refuse destroy without <code>--force</code>. Service bindings decide deploy order. Queue consumers are removed before workers on teardown. Missing secrets block deploy before you get half a release.</p>
  </div>
  <div class="feature-cell">
    <h3>More visibility, less guessing</h3>
    <p><code>wd plan</code> shows what a stage will create. <code>wd status</code> shows what drifted. <code>wd verify</code> checks rendered config, workers, bindings, and secrets after deploy. It is much easier to explain what the tool is doing because it tells you.</p>
  </div>
</div>

<span class="section-label">Zero-codegen types</span>
<h2 class="section-title">Declare resources once. Types flow everywhere.</h2>
<p class="section-desc">
  wrangler-deploy infers your Worker <code>Env</code> type directly from the resources you declare in config. No build step, no generated files, no type-sync commands. Rename a binding and TypeScript catches every mismatch across every worker.
</p>

</p>

<div>
  <span class="section-label">Get started</span>
  <h2 class="section-title">Start a new app or adopt the Wrangler project you already have.</h2>
  <div class="install-cmd">
    <span class="dollar">$</span> npm install -D wrangler-deploy
  </div>
  <div class="gap-4">
    <p><strong>Repo already has <code>wrangler.jsonc</code>?</strong> Run <code>wd init</code> to generate project config from local files without changing your JSONC.</p>
    <p><strong>Already deployed in Cloudflare?</strong> Run <code>wd introspect</code> to generate config from the live account.<br/></p>
    <p>Then run <code>wd apply --stage staging</code> and <code>wd deploy --stage staging</code>. Local dev stays on Wrangler and your checked-in JSONC stays the authoring source. Stage deploys get rendered JSONC with the right IDs.</p>
  </div>
