<p class="section-desc">
  Most Workers projects already have <code>wrangler.jsonc</code>. The problem starts when one worker turns into several workers, shared resources, and per-branch environments.

<span class="section-label">What you get</span>
<h2 class="section-title">A better deploy workflow without throwing away your Wrangler workflow.</h2>

<div class="feature-rows">
  <div class="feature-cell">
    <h3>JSONC stays first-class</h3>
    <p>Your existing <code>wrangler.jsonc</code> files stay in the repo and stay readable. No rewrite step, no migration away from Wrangler. <code>wd init</code> reads them, <code>wrangler dev</code> still uses them, and <code>wd deploy</code> uses rendered JSONC built for the target stage.</p>
  </div>
  <div class="feature-cell">
    <h3>One tool for dev and deploy</h3>
    <p>Keep using Wrangler locally, or run <code>wd dev</code> to start every worker together with sane ports and dependency order. Then use the same project definition to apply, deploy, verify, and destroy full stages.</p>
  </div>
  <div class="feature-cell">
    <h3>Types without codegen</h3>
    <p>Import your Env type directly from the config file. <code>D1Database</code>, <code>KVNamespace</code>, <code>Queue</code>, and service binding types resolve at compile time. No generated files to commit.</p>
  </div>
  <div class="feature-cell">
    <h3>Shared state across the team</h3>
    <p>Store state in a Cloudflare KV namespace and stop relying on one laptop having the latest stage metadata. One person can apply, another can deploy, CI can verify or destroy, and everyone sees the same state.</p>
  </div>
  <div class="feature-cell">
    <h3>Safer staged deploys</h3>
    <p>Protected stages refuse destroy without <code>--force</code>. Service bindings decide deploy order. Queue consumers are removed before workers on teardown. Missing secrets block deploy before you get half a release.</p>
  </div>
  <div class="feature-cell">
    <h3>More visibility, less guessing</h3>
    <p><code>wd plan</code> shows what a stage will create. <code>wd status</code> shows what drifted. <code>wd verify</code> checks rendered config, workers, bindings, and secrets after deploy. It is much easier to explain what the tool is doing because it tells you.</p>
  </div>
</div>

<div class="cta-section">
  <span class="section-label">Get started</span>
  <h2 class="section-title">Start from the Wrangler project you already have.</h2>
  <div class="install-cmd">
    <span class="dollar">$</span> npm install -D wrangler-deploy
  </div>
  <p class="section-desc" style="margin-inline: auto; text-align: center;">
    <strong>Repo already has <code>wrangler.jsonc</code>?</strong> Run <code>wd init</code> to generate project config from local files.<br/>
    <strong>Already deployed in Cloudflare?</strong> Run <code>wd introspect</code> to generate config from the live account.<br/>
    Then run <code>wd apply --stage staging</code> and <code>wd deploy --stage staging</code>. Local dev stays on Wrangler. Stage deploys get rendered JSONC with the right IDs.
  </p>
</div>
