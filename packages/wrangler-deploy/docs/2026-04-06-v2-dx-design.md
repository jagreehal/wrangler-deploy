# wrangler-deploy v2 — DX Design Spec

> The goto library for Cloudflare Workers. From solo dev to team, from local dev to production.

## Goals

Make `wrangler-deploy` indispensable for any Cloudflare Workers project by delivering:

1. Full visibility into what depends on what
2. Orchestrated local development that just works
3. First-class CI/CD with PR preview environments
4. Gradual config management that respects `wrangler.jsonc` as the source of truth
5. Typed Env without codegen, layered on top of wrangler config

`wrangler.jsonc` is king. `wrangler-deploy` makes everything around it excellent.

## Audience

Solo devs and teams equally. The tool scales from one worker to a monorepo of twenty, from `wd dev` on a laptop to `wd deploy --stage pr-42` in CI.

## Phase 1: Graph & Visibility

### `wd graph [--stage <name>]`

Print the dependency topology to the terminal. Workers are nodes. Edges represent: service bindings, queue producer→consumer→DLQ chains, and shared resources (Hyperdrive, D1, KV bound to multiple workers).

Without `--stage`, shows the config-level topology. With `--stage`, overlays live resource IDs and status from state.

Output is color-coded by type: workers, KV, D1, queues, Hyperdrive, R2, Vectorize.

### `wd graph --format mermaid|dot|json`

Machine-readable output for different contexts:

- **mermaid** — paste into GitHub PRs, docs, or any Markdown renderer
- **dot** — Graphviz DOT for custom rendering
- **json** — structured graph model for tooling and scripts

### `wd impact <worker-path>`

Walk the graph upstream and downstream from a given worker. Output is actionable, not just a list:

```
$ wd impact apps/batch-workflow

  Upstream (depends on):
    hyperdrive → shared with apps/api, apps/event-router
    cache-kv → exclusive

  Downstream (depended on by):
    apps/api → WORKFLOWS service binding

  If apps/batch-workflow is unavailable:
    apps/api loses WORKFLOWS binding (service calls will fail)
    apps/event-router is unaffected (no direct dependency)
```

### `wd diff <stage-a> <stage-b>`

Compare two stages side by side:

- Resources: exists in A but not B, different IDs, drift
- Workers: version mismatches, different routes
- Secrets: set in A but missing in B
- Config: binding differences

Table output by default. `--format json` for CI consumption.

### Implementation approach

The graph engine in `graph.ts` already does topological sort and cycle detection. Extend it to produce a rich graph model: nodes carry metadata (type, resource IDs, status), edges carry relationship types (service-binding, producer, consumer, dead-letter, shared-resource). Renderers (ASCII tree, Mermaid, DOT, JSON) consume the model through a shared interface.

`wd impact` walks the graph in both directions from a given node. `wd diff` loads two stage states and diffs them structurally.

## Phase 2: `wd dev` — Orchestrated Local Development

### Core behavior

`wd dev` reads `wrangler-deploy.config.ts`, resolves the dependency graph, and starts all workers via `wrangler dev` in dependency order. Each worker gets an assigned port. Service bindings between workers are wired via local HTTP using wrangler's `--port` flag and localhost service binding resolution.

### Port management

Default base port 8787, incrementing per worker in dependency order. Override per worker in config:

```ts
worker("apps/api", { devPort: 8787 })
```

Port map printed to terminal on startup:

```
  apps/batch-workflow → localhost:8788
  apps/event-router   → localhost:8789
  apps/api            → localhost:8787
```

### Local resources

- **D1** — wrangler's built-in local mode with `.wrangler/state` persistence. If a `migrations/` directory exists in the worker path, `wd dev` applies migrations automatically on startup.
- **KV** — local persistence via wrangler's built-in local mode. Optional seed data via config: `kv("cache", { devSeed: { "key": "value" } })`.
- **Queues** — wrangler's local queue simulation between co-located workers. `wd dev` ensures producer and consumer workers are both running so messages flow end to end.
- **Hyperdrive** — reads `localConnectionString` from the worker's `.dev.vars` or `wrangler.jsonc`. Validates Postgres connectivity on startup. If not found, warns with instructions rather than failing silently.
- **R2** — local persistence via wrangler's built-in local mode.

### Hot-reload topology

`wd dev` leverages wrangler's native file watching for individual worker restarts. On top of that:

- When `wrangler-deploy.config.ts` changes, the entire topology re-evaluates. New workers start, removed workers stop, port assignments update.
- When a shared workspace package changes (detected via `package.json` workspace dependencies), all workers that depend on it restart.

### Terminal UI

- Split log output per worker, color-coded by worker name
- Status line showing which workers are running, their ports, and health
- Errors from one worker don't bury output from others

### Filtered dev

```
wd dev --filter apps/api
```

Start only the specified worker plus its transitive dependencies (resolved from the graph). Useful when working on a single service without needing the full topology running.

### Pass-through to wrangler

`wd dev` passes through to `wrangler dev` for each worker using their existing `wrangler.jsonc`. It does not generate or modify wrangler configs for local dev. Additional wrangler flags can be specified per worker:

```ts
worker("apps/api", { devArgs: ["--inspect"] })
```

## Phase 3: CI/CD — GitHub Actions & PR Integration

### `wd ci init --provider github`

Generates `.github/workflows/wrangler-deploy.yml` with three triggers:

- **PR opened/synchronized** — apply resources → deploy workers → verify → comment on PR
- **PR closed/merged** — destroy the PR stage
- **Push to main** — deploy to production (skip apply if resources exist)

The generated workflow is a starting point that users own. No hidden runtime dependency on `wd` internals — the workflow calls the same CLI commands a developer would.

### `wd ci comment`

After deploy, post or update a single comment on the PR containing:

- **Live URLs** for each deployed worker (routes, custom domains, or `.workers.dev`)
- **Mermaid topology diagram** (renders natively in GitHub Markdown)
- **Resource summary table** — what was created, what was reused, IDs
- **Verification results** — pass/fail per worker with details
- **Diff from previous deploy** — what changed since last push to this PR
- **Secrets status** — all set vs. missing (names only, never values)

Updates the same comment on subsequent pushes instead of posting new ones. Identified by a hidden marker in the comment body.

### `wd ci check`

Post GitHub check runs via the API:

- **`wrangler-deploy/apply`** — did resources provision successfully?
- **`wrangler-deploy/deploy`** — did all workers deploy without errors?
- **`wrangler-deploy/verify`** — post-deploy coherence: health checks pass, bindings valid, secrets present

Each check includes detailed output in the GitHub checks UI — not just pass/fail but structured results.

### Secrets in CI

- `wd secrets sync --to pr-$N --from-env-file .dev.vars.ci` for bulk injection
- `wd secrets set --stage pr-$N --from-env KEY=$VALUE` for piping from CI secret stores

### Provider abstraction

GitHub Actions is first-class. The CI layer uses a thin `CiProvider` interface so GitLab CI, Bitbucket Pipelines, and others can be added later. Generated workflow files are provider-specific. `wd ci comment` and `wd ci check` work through the abstraction using environment detection (e.g., `GITHUB_TOKEN`, `CI_MERGE_REQUEST_IID`).

## Phase 4: Gradual Config Takeover

### Principle

`wrangler.jsonc` is always the source of truth. When users opt in to managed mode, `wd` writes resource binding IDs directly into `wrangler.jsonc` under the appropriate environment section — not into shadow files. The wrangler file remains authoritative for `wrangler dev`, `wrangler deploy`, and all other tooling.

### Opt-in per worker

```ts
worker("apps/api", { managed: true })
```

- **`managed: false`** (default) — current behavior. Rendered configs go to `.wrangler-deploy/<stage>/`. Existing wrangler.jsonc untouched.
- **`managed: true`** — `wd apply` writes resource binding IDs into the worker's `wrangler.jsonc` under a stage-named environment section. `wd` automates the manual copy-paste-IDs workflow.

### What managed mode controls

- Resource binding IDs: KV namespace IDs, D1 database IDs, queue names, Hyperdrive config IDs
- Service binding targets updated when worker names change per stage
- Nothing else. Compatibility dates, build commands, routes, module rules, and all other config stays untouched.
- JSONC comments in the wrangler file are preserved. `wd` uses a JSONC-aware parser/serializer (already exists in `wrangler.ts`) that round-trips comments.

### Typed Env alongside wrangler.jsonc

The phantom type system (`workerEnv`, `kv()`, `d1()`, etc.) works alongside wrangler.jsonc:

- `wrangler-deploy.config.ts` declares bindings for type safety
- `wrangler.jsonc` holds the real resource IDs
- Worker code imports: `type Env = typeof api.Env`
- TypeScript derives the full Env type at compile time

No codegen. No build step. No generated files in the type chain.

### Validation

`wd verify` gains a new check for managed workers: confirm that `wrangler.jsonc` binding IDs match state. If someone hand-edits a binding ID, `wd verify` flags the drift with a clear message and fix command.

### Migration path

1. Start with `managed: false` — zero changes to existing workflow
2. Flip to `managed: true` per worker when ready
3. `wd apply` handles the transition — writes IDs into `wrangler.jsonc` on first managed apply
4. Fully reversible — set back to `false`, binding IDs remain in `wrangler.jsonc`, `wd` stops updating them

## Cross-Cutting Concerns

### `wd doctor`

Diagnostic command that checks everything:

- wrangler installed and authenticated (`wrangler whoami`)
- Config file valid and parseable
- All worker paths exist with valid `wrangler.jsonc`
- Dependency graph has no cycles
- For a given stage: state consistent, resources exist, no drift, secrets set

First step for debugging any issue. Output is structured: pass/warn/fail per check with remediation instructions.

### Error messages

Every error includes three things: what failed, why, and what to do next. No stack traces unless `--verbose`.

```
✗ Hyperdrive 'payments-db-pr-42' creation failed: no --database-url provided
  Run: wd apply --stage pr-42 --database-url <your-postgres-url>
```

### Timing

All commands print wall-clock duration. `wd apply` and `wd deploy` show per-resource and per-worker timing to help users optimize CI pipelines. No external telemetry.

### `wd status` improvements

Extend current status output to show:

- Resource table: IDs, creation dates, drift status
- Worker deploy status: version hash, last deployed, routes
- Topology summary: compact inline graph
- Secrets: set/missing per worker

### Shell completions

```
wd completions --shell zsh|bash|fish
```

Generate shell completion scripts. Stage names autocomplete from the state directory. Worker paths autocomplete from config. Commands and flags autocomplete statically.

### Config validation

On every command, validate config before execution. Catch early:

- Duplicate resource logical names
- Invalid or non-existent worker paths
- Circular service bindings
- Queue consumers without a matching producer
- Dead-letter queues referencing non-existent queues
- Bindings referencing undefined resources

Fail fast with clear, specific messages.

---

## What does not change

- `wrangler.jsonc` remains the source of truth for worker configuration
- `wrangler dev` and `wrangler deploy` continue to work independently
- All Cloudflare operations go through `wrangler` CLI (no direct API calls for resource CRUD)
- State backend supports local files and remote KV
- Stage protection and TTL rules work as today
- The phantom type system is additive — never required, always available
