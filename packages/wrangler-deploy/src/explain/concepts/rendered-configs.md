# rendered-configs

**Rendered configs** are the per-worker, per-stage `wrangler.jsonc` files
that `wd apply` writes to `.wrangler-deploy/<stage>/<workerPath>/wrangler.jsonc`.
They are what `wrangler deploy` actually reads — your hand-written
`wrangler-deploy.config.ts` never gets shipped.

What rendering does:

1. Resolves the worker's bindings against the stage's provisioned resources
   (e.g. fills in the real `database_id` for a D1 binding).
2. Applies any stage-specific overrides declared in your config.
3. Injects secrets references (the actual values stay in `state.json`, encrypted).
4. Stamps a `# generated by wrangler-deploy` header so audits can tell.

**Never edit these files by hand.** They will be overwritten on the next
`wd apply`. If you need to change them, change the declared config and
re-apply. If you suspect drift, run `wd env diff --stage <name>` to compare
the local + rendered shapes.

If `wd deploy` finds a rendered config older than the corresponding
`state.json` (or missing entirely), it emits `WD_E_RENDERED_CONFIG_STALE`
and points you at `wd apply` to refresh.

- Concepts: `wd explain bindings`, `wd explain state`, `wd explain stages`
- Commands: `wd apply`, `wd deploy`, `wd env diff`
- Errors: `wd explain WD_E_RENDERED_CONFIG_STALE`
